# Оверхед асимметричного OpenPGP-шифрования: можно ли ужать?

Дата: 2026-07-17. Среда: Node v22.14.0, openpgp.js 6.3.1, GnuPG 2.2.27 / libgcrypt 1.9.4 (Linux 5.15).
Скрипт: `measure-overhead.mjs` (генерирует ключи всех типов, шифрует payload'ы 16–1024 Б,
расшифровывает системным `gpg` в изолированном `--homedir`, печатает `--list-packets` и таблицу размеров).

## Из чего состоит бинарное сообщение (наша текущая структура)

```
PKESK (tag 1)  — сеансовый ключ, зашифрованный ключом получателя
SEIPD (tag 18) — симметричный контейнер (CFB + MDC), внутри:
  [Compressed (tag 8), если zip] → Literal (tag 11) → payload
```

Замеренный фиксированный оверхед (aes128, без сжатия, incompressible payload):

| Ключ получателя | PKESK, байт | Полный оверхед, байт | ≈ предложений (20 бит/шт) |
|-----------------|-------------|----------------------|---------------------------|
| Curve25519      | 80          | **131–133**          | ~53                       |
| NIST P-256      | 112         | 163–165              | ~66                       |
| RSA-2048        | 271         | 322–324              | ~129                      |
| RSA-3072        | 399         | 450–452              | ~180                      |

Симметричная «обёртка» (SEIPD + Literal + MDC) при aes128 без сжатия — **51 байт** независимо от ключа:
SEIPD-заголовок 2 + версия 1 + CFB-префикс 18 + Literal-заголовок 8 + MDC-пакет 22
(проверка: Curve25519, payload 16 Б → 80 + 51 + 16 = 147 Б, совпадает с замером).
Это фактически пол спецификации RFC 4880 — ужимать больше нечего, не ломая формат.

## Что проверено и что даёт

| Рычаг | Экономия | Совместимость / стойкость | Вердикт |
|-------|----------|---------------------------|---------|
| aes128 вместо aes256 (для ECDH-ключей) | 16 Б в PKESK | ok / 128 бит достаточно | **уже используется** |
| Отключить zip для несжимаемых данных | 6–9 Б | ok | **включено адаптивно в `gpg-crypto.js`** |
| Оставить zip для сжимаемого текста | до −790 Б на 1 КиБ текста | ok | оставить |
| Убрать MDC (tag 9 вместо 18) | 23 Б | gpg ≥2.2.8 отказывается расшифровывать; теряется целостность | **нет** |
| SEIPD v2 / AEAD (OCB) | не меньше (чанк-теги) | gpg 2.2 не поддерживает | нет |
| Скрытый получатель (wildcard keyid) | 0 Б (поле фиксированное) | — | нет смысла для размера |
| Обнулить дату в Literal | 0 Б (поле фиксированное 4 Б) | — | нет |
| Ключ Curve25519 вместо RSA | **190–320 Б** | ok, gpg ≥2.1 | главный рычаг, но это выбор получателя |

## Выводы

1. Симметричная часть уже на минимуме формата OpenPGP (49 Б оверхеда). Единственная локальная
   оптимизация — **адаптивное сжатие**: шифровать и с zip, и без, брать меньший результат
   (сейчас zip включён всегда и тратит 6–9 Б на несжимаемых данных). Оба варианта штатно
   читаются gpg/Kleopatra.
2. Доминирует PKESK, а его размер задан алгоритмом ключа получателя. Curve25519 экономит
   ~190 Б против RSA-2048 и ~320 Б против RSA-3072 (это 76–128 предложений cover-текста).
   Повлиять можно только рекомендацией: просить получателей давать Curve25519-подключ.
3. Всё остальное (убрать MDC, AEAD-контейнеры, скрытый keyid) либо ломает совместимость
   с gpg 2.2 / снижает стойкость, либо не даёт байт.

Проверка: каждый вариант из таблицы измерений расшифрован системным `gpg --decrypt`
с побайтовым сравнением plaintext (128 комбинаций, все прошли).
